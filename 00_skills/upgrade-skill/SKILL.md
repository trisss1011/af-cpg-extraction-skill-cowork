# CPG 마스터 업그레이드 스킬

## 목적

cpg-data-extraction의 신 버전이 나올 때마다 기존 마스터 엑셀을 최신 사양으로 안전하게 업그레이드한다. 본 스킬은 **단일 트리거**(작업자가 "업그레이드 해줘" 한 마디)로 작동하며, 마스터의 현재 버전을 자동 감지해 필요한 모든 변경을 순차로 적용한다.

대상은 작업자 본인의 단일 마스터 파일(`90_Output/AF_CPG_data_extraction_<본인이름>.xlsx`). 추출 파일(`extracts/AF_extract_*.xlsx`)에는 적용하지 않는다.

---

## 작업 흐름

```
마스터 식별 → 락 파일 확인 → _meta 시트 로드/생성 → 현재 spec_version 확인 →
백업 → 적용할 버전 목록 산출 (예: 2.5.x→2.6, 2.6→2.6.1) →
버전별 변경 순차 적용 (references/v<버전>_changes.md) →
미리보기 출력 → 사용자 승인 →
변경 적용 + _meta 갱신 + 저장 →
사람 확인 항목 → 채팅창 인터랙티브 처치 (카테고리별 순회, 작업자 응답대로 마스터 즉시 수정) → 리포트 .md 병행 저장 → 결과 보고
```

---

## 핵심 원칙

1. **단일 트리거 + 자동 버전 감지**: 작업자는 어느 버전에서 멈춰 있는지 신경 쓰지 않는다. `_meta` 시트의 `spec_version`을 보고 자동으로 필요한 변경만 적용.
2. **멱등성**: 이미 최신 버전이면 변경 0건으로 종료. 같은 마스터에 두 번 실행해도 안전.
3. **백업 우선**: 변경 시작 전 `90_Output/backups/master_before_upgrade_<YYYYMMDD_HHMMSS>.xlsx` 자동 생성.
4. **원자성**: 메모리에서 모든 변경 처리 후 단일 `save()`. 중간 실패 시 백업으로 복구.
5. **자동 vs 수동 분리**: 기계적으로 결정 가능한 항목만 자동(미리보기 → "예" 일괄 적용). 모호한 항목은 자동 변경 절대 금지, 8단계 인터랙티브 처치로 넘어가 작업자 응답에 따라 처치.
6. **사용자 승인 + 인터랙티브 처치**: ① 자동 처리는 미리보기 → 명시적 "예"만 진행 (7단계). ② 사람 확인 항목은 카테고리별로 채팅에 표시 → 작업자 응답대로 마스터에 즉시 반영 (8단계). 둘 다 자동 진행 금지.
7. **버전 명세는 references로 분리**: SKILL.md는 흐름만, 각 버전 룰은 `references/v<버전>_changes.md`로 분리해 확장성 확보.
8. **`_meta` 시트는 본 스킬 전용**: 작업자가 직접 편집하지 않는다. 본 스킬만 read/write.
9. **인터랙티브 처치 안전망**: 8단계 시작 시 추가 백업 1회 (`master_before_phase8_<시각>.xlsx`). 카테고리 단위 단일 save. 행 삭제는 역순. 셀 서식 보존(merge-skill 알고리즘 재사용).

## `_meta` 시트 명세

마스터 엑셀 안에 숨김 시트 `_meta`를 두어 업그레이드 이력을 추적한다.

### 구조

| key | value |
|---|---|
| spec_version | `2.6.1` (예) |
| last_upgrade | `2026-05-20` (예) |
| last_actor | `upgrade-skill v1.2` (예) |
| created_at | `2026-04-04` (예) |

- 시트 첫 행은 헤더 `key | value`
- 한 키당 한 행
- 미래 확장을 위해 wide form 대신 key-value 구조 사용
- 시트 hidden 속성 적용 (`ws.sheet_state = "hidden"`) — 작업자가 일반 사용 중 보이지 않음

### `spec_version` 키 의미와 하위 호환 (v1.2 변경)

- **의미**: 이 마스터가 적용받은 `cpg-data-extraction` **사양(specification) 버전**. 열 구조(헤더 구성)와 의미·기재 규칙을 모두 포함하는 "스킬 사양"의 식별자.
- **명명 이력**: v1.1까지는 `schema_version`이었으나, v2.6 → v2.6.1처럼 열 구조 변경 없이 규칙만 보강된 케이스에서 "스키마(schema)"라는 용어가 부적절하다는 판단에 따라 v1.2에서 **`spec_version`으로 리네임**.
- **하위 호환**: 본 스킬은 마스터를 읽을 때 두 키를 모두 인식한다. 우선순위: `spec_version` → `schema_version`. 둘 다 있으면 `spec_version` 사용. `schema_version`만 있는 기존 마스터는 본 스킬 실행 시 `spec_version` 키로 자동 마이그레이션(값 복사 후 구 키 삭제) + `last_upgrade` 갱신.
- 마이그레이션은 자동 처리 항목으로 미리보기에 1줄 표시:
  ```
  _meta 키 마이그레이션: schema_version → spec_version (1건)
  ```

### `_meta` 자동 생성 규칙

- `_meta` 시트가 없으면 본 스킬이 처음 실행될 때 자동 생성
- 헤더 컬럼 구성으로 현재 버전 추정 후 spec_version 초기값 결정:
  - 기본정보 47열, 끝이 `notes` → `2.5` (정확히는 2.5.x — 첫 업그레이드에서 2.6 → 2.6.1로 순차 진입)
  - 기본정보 48열, AU=`analysis_set`, AV=`notes` → `2.6` (다음 업그레이드에서 2.6.1로 진입)
  - v2.6.1은 열 구조 변경이 없으므로 헤더만으로는 2.6 vs 2.6.1 구분 불가. `_meta`가 없으면 보수적으로 `2.6`으로 잡고, upgrade-skill 실행 시 2.6 → 2.6.1 변경(주로 사람 확인 항목)을 한 번 더 통과시킨다 (멱등성).
  - 추후 버전은 동일 패턴으로 추정 규칙 추가

## 1단계: 마스터 파일 식별

- 기본: `90_Output/AF_CPG_data_extraction_<본인이름>.xlsx`
- 후보가 다수면 사용자에게 선택 질문
- 템플릿 파일명(`AF_CPG_data_extraction_master_[작업자이름].xlsx`)은 우선순위 낮춤

## 2단계: 락 파일 확인

`~$<마스터파일명>` 존재 시 즉시 중단.

## 3단계: `_meta` 시트 처리 + 현재 버전 식별

- `_meta` 시트 없으면 자동 생성, 헤더 구성으로 spec_version 추정
- 있으면 그 값 그대로 사용
- 결과로 `current_version` 확보 (예: `2.5` 또는 `2.6`)

## 4단계: 백업

`90_Output/backups/master_before_upgrade_<YYYYMMDD_HHMMSS>.xlsx`로 `shutil.copy2`.

## 5단계: 적용할 변경 목록 산출

본 스킬이 지원하는 최신 버전을 `LATEST_VERSION` 상수로 보유 (현재 `2.6.1`).

- `current_version >= LATEST_VERSION`: 변경 없음 → 미리보기에 "이미 최신" 표시 후 정상 종료
- `current_version < LATEST_VERSION`: 적용할 버전 목록을 순서대로 산출 (예: current=`2.5` & latest=`2.6.1` → `[2.6, 2.6.1]`)

각 버전 변경 명세는 `references/v<버전>_changes.md`에서 읽어 처리.

2. v2.6.1 변경 (`references/v2.6.1_changes.md`):
   - 자동: `_meta` 키 마이그레이션(`schema_version` → `spec_version`), `spec_version` 갱신(2.6.1)
   - 사람 확인 (8단계 인터랙티브): KM vs WM 단독비교 행 제거 후보, S열 코드 5 케이스 세분화 재검토, T열 자유 텍스트 누락 후보, AN/AU 보수적 판정 재분류(추정 PP/ITT → NR 후보), FAS 명시 논문의 코드 재분류, **미네소타 QOL 점수 방향 모순 검출**(MLHFQ → QOL(Minnesota cited))

## 7단계: 사용자 승인 + 적용

"예" 답변에만 변경을 마스터에 반영. 다른 답은 중단(백업은 유지).

`_meta` 시트의 `spec_version`을 `LATEST_VERSION`으로 갱신, `last_upgrade` 오늘 날짜, `last_actor` 본 스킬 버전 기록.

행 수 검증 후 단일 `save()`. 사후 검증(헤더·핵심 변경)으로 무결성 확인. 실패 시 백업 자동 복구.

## 8단계: 인터랙티브 사람 확인 항목 처리 (v1.2 개정)

자동 처리가 끝나면, 사람 확인 항목을 **채팅창에서 카테고리별로 순차 처치**한다. cpg-data-extraction의 6B 채팅 출력(추출 결과를 한 항목씩 보여주고 작업자가 확인·수정)과 같은 인터랙티브 흐름.

### 8.1 카테고리 요약 출력

전체 사람 확인 항목을 **카테고리별 카운트 표**로 한 번에 보여주고, 어디서부터 시작할지 작업자에게 묻는다.

```
사람 확인 항목 — 인터랙티브 처치
─────────────────────────────────
[v2.6]
  A. study_design 비표준 값:        N건
  B. quasi-RCT 재분류 후보:        N건
  C. af_type_other 코드 5 부적격:   N건
  D. SAE 추가 후보:                 N건
  E. HRV 모호 매치 (자동 보류):      N건
[v2.6.1]
  F. AU 추정값(보수적 위반):         N건
  G. FAS 승격 후보(ITT→FAS):         N건
  H. KM vs WM 단독비교 제거 후보:     N건
  I. S열 코드 5 케이스 재검토:        N건
  J. T열 af_type_text 누락:          N건
  K. 미네소타 QOL 변환 후보:          N건
  L. 3arm 비교쌍 점검:                N건

총 N건.
어디부터 처리할까요?
- "순서대로" : A부터 L까지 차례로
- "F부터" / "H만" / "A,C,H" : 원하는 카테고리만
- "지금은 안 함" : 8단계 종료, 리포트 .md만 저장
```

### 8.2 카테고리별 인터랙티브 처치

작업자가 선택한 카테고리의 항목을 **번호 매긴 표로** 채팅창에 출력한다. 각 항목은 1행 1줄 원칙(추출 6B 출력 방식 재사용). 항목당 표시: `번호 / study_id / 행번호 / 현재값 / 제안값 / 원문 대조 포인트`.

예시 (H. KM vs WM 단독비교 제거 후보):

```
[H. KM vs WM 단독비교 제거 후보] 5건
─────────────────────────────────
1. study_id=Zhang_2019 (기본정보 행 5)
   comparison_type: KM_alone_vs_WM
   같은 study_id에 다른 비교쌍 없음 → 2arm 단독 연구
   제안: exclude=Y + exclude_reason="v2.6.1: KM vs WM 단독비교 제외"

2. study_id=Yao_2012 (기본정보 행 12 + 13)
   comparison_type: KM_alone_vs_WM (행 12), KM+WM_vs_WM (행 13)
   3arm 연구로 추정 → 행 12만 제거, 행 13 보존
   제안: 행 12 삭제

3. study_id=Wang_2024 (기본정보 행 18)
   comparison_type: KM_alone_vs_WM
   ⚠️ 원문 재확인 필요 — 표준치료 병용 여부 불명
   제안: 작업자 판단

...

처치 방식?
- "전부 제안대로 적용" : 5건 모두 자동 처치
- "1,2 적용, 3 건너뛰기" : 개별 선택
- "1번 원문 보기" : 해당 행의 AN/AD/AI 원문 발췌 추가 표시
- "이 카테고리 건너뛰기" : H 통째로 미처리
- "마침" : 처리 중단, 남은 항목은 .md 리포트로 보존
```

### 8.3 즉시 적용 + 단일 save

작업자 응답대로 **마스터에 변경을 즉시 반영**한다 (안내만 하는 게 아니라 스킬이 실제 셀 수정·행 삭제·exclude 마킹 등 수행).

- 안전: 카테고리 처치 시작 직전에 추가 백업 1회 (`master_before_phase8_<시각>.xlsx`)
- 메모리에서 모든 변경 누적 후 카테고리 완료 시점에 단일 `save()`
- 행 삭제는 `ws.delete_rows`로 역순 처리, 인덱스 어긋남 방지
- 셀 서식·배경색·테두리 전부 보존 (merge-skill의 서식 보존 알고리즘 재사용)

### 8.4 다음 카테고리로 진행

한 카테고리 끝나면 ✓ 표시 후 다음 카테고리로. 작업자가 "마침" 선언하거나 모든 카테고리 처치 끝나면 8단계 종료.

처치 결과 요약 출력 예시:
```
처치 완료 — 진행 상황
─────────────────────────────────
✓ H. KM vs WM 단독비교 제거: 4건 적용 / 1건 건너뜀
✓ G. FAS 승격: 3건 적용
   F. AU 추정값: 미처리 (12건 남음 — 리포트 참조)
   ...
```

### 8.5 종합 리포트 .md (병행 저장)

8단계 진행 중 모든 항목(처치 적용분 + 건너뛴 분 + 미처리 분)을 `90_Output/retrofit_report_<YYYYMMDD_HHMMSS>.md`에 누적 저장한다. 세션이 끊겨도 리포트로 이어서 처치 가능.

리포트 구조:
- 헤더: 실행 시각, current→latest 버전, 적용한 자동 처리 카운트
- 카테고리별 표: 항목 / 적용 여부 / 작업자 응답 / 마스터 변경 내역
- 미처리 항목 목록: 다음 세션에서 재실행 시 우선 처리 권장

## 9단계: 결과 보고

- 적용한 버전(들), 자동 처리 카운트, 인터랙티브 처치 카운트(적용/건너뜀/미처리)
- 백업·리포트 파일 경로
- 미처리 항목이 남아있으면 "본 스킬 다시 실행하면 미처리 항목부터 재개됩니다" 안내
- 모든 항목 처치 완료 시 멱등성 확인 권장 (한 번 더 실행 → "변경 0건" 응답이면 안정)

## 절대 금지

- **사용자 승인 없는 적용**: 미리보기 "예" 없으면 변경 금지
- **백업 없이 마스터 수정**: 백업 실패는 중단 사유
- **추출 파일 수정**: extracts/ 안 어떤 파일도 건드리지 않음
- **다른 작업자 마스터 처리**: `AF_CPG_data_extraction_<다른 사람>.xlsx` 실행 금지
- **모호 항목의 자동 변경**: 각 버전 명세의 "사람 확인" 항목은 자동 변경 절대 금지
- **`_meta` 시트 임의 수정**: 본 스킬만 read/write. 작업자가 직접 편집하지 않음
- **AI 단독 SAE 정의 적용**: 논문이 SAE로 묶어 보고하지 않은 항목을 임의로 SAE 행에 합산 금지
