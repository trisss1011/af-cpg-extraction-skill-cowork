# v2.6.1 변경 명세 (cpg-data-extraction)

> 본 파일은 upgrade-skill이 마스터를 v2.6 → v2.6.1로 업그레이드할 때 적용하는 변경 항목을 정의한다.
> 버전 식별: `_meta` 시트의 `spec_version` 값이 `2.6.1` 미만이면 본 파일의 변경을 적용.
> 적용 후 `_meta`에 `spec_version=2.6.1`, `last_upgrade=<YYYY-MM-DD>` 기록.

---

## v2.6.1 변경의 성격

v2.6.1은 **열 구조 변경이 없는 의미·기재 규칙 보강** 릴리스다. 따라서:
- 자동 처리 가능한 항목은 매우 제한적 (`_meta` 키 마이그레이션 + `spec_version` 갱신만)
- 핵심 변경은 사람 확인 항목으로 처리 — 원문 대조 필요
- 멱등성 보장: 같은 마스터를 두 번 실행해도 안전

기존 v2.6 마스터는 헤더만으로는 2.6 vs 2.6.1 구분이 불가능하므로, `_meta`가 비어 있으면 보수적으로 `2.6`으로 잡고 본 명세를 한 번 통과시킨다.

## 자동 처리 룰 상세

### 1. `_meta` 키 마이그레이션 (v1.2 명명 변경)

- 기존 `schema_version` 키가 있고 `spec_version` 키가 없으면:
  - `spec_version`에 `schema_version` 값을 복사
  - `schema_version` 키 삭제
- 둘 다 있으면 `spec_version` 사용, `schema_version` 삭제 (중복 제거)
- 둘 다 없으면 v2.6 명세의 헤더 구성 추정 규칙으로 `spec_version` 초기값 결정
- 멱등: 이미 `spec_version`만 있으면 skip

### 2. spec_version 갱신

- `_meta` 시트의 `spec_version` 값을 `2.6.1`로 덮어쓴다.
- `last_upgrade`에 오늘 날짜(`YYYY-MM-DD`), `last_actor`에 `upgrade-skill v1.2` 기록.
- `_meta` 시트가 없으면 v2.6 명세의 자동 생성 규칙으로 먼저 생성 후 본 갱신 적용.

> **참고 (v2.6.1 합의)**: 미네소타 QoL 관련 일괄 자동 명칭 변환 룰은 **두지 않는다**. 표준 MLHFQ로 추출된 행 중 **점수 방향이 모순되는 케이스만** `QOL(Minnesota cited)`로 변환하는데, 이 판정은 원문 대조가 필요해 8단계 §F 인터랙티브로만 처리. 새 논문 추출 시점에는 cpg-data-extraction이 의심 케이스를 6B ⑤로 띄워 추출자가 처음부터 `QOL(Minnesota cited)`로 적도록 안내.

## 사람 확인 항목 검출 룰 (자동 변경 X)

### A. AU `analysis_set` 추정값 검출 — 보수적 판정 위반 후보

v2.6.1에서 AU는 **method에 명시된 것만** 코드화한다. v2.6 시절 "탈락자 제외" 서술로 PP 추정된 행이나, 결과 표 분모만으로 ITT 추정된 행을 검출.

**검출 알고리즘**:
1. 각 행의 AN(`rob_d2_analysis`)을 읽는다.
2. AN이 `NR`이거나, AN에 ITT/PP/mITT/FAS 키워드(아래 화이트리스트)가 정확히 매치되지 않는데 AU가 `ITT`/`PP`/`FAS` 중 하나면 후보.
3. 화이트리스트 (정확 매치 + 부정 키워드 윈도 검사, v2.6의 §2 알고리즘 재사용):
   - ITT: `ITT`, `intention-to-treat`, `의도-치료군`, `의도치료군`
   - mITT: `mITT`, `modified ITT` → ITT로 통합
   - PP: `PP`, `PPS`, `per-protocol`, `per protocol`, `프로토콜순응군`
   - FAS: `FAS`, `full analysis set`, `Full Analysis Set`

**8단계 채팅 표시 항목**: study_id, 행번호, AN 원문, 현재 AU 값, 제안값(`NR`)

**작업자 처리**: 원문 대조 후 NR로 변경하거나, method에 실제 명시가 있다면 AN 원문을 보강.

### B. FAS 명시 논문 재분류 후보 — `ITT` → `FAS` 승격

v2.6에서 FAS는 ITT로 통합 코드화되었다. v2.6.1에서 FAS가 정식 코드로 분리되었으므로, AN에 FAS가 명시된 행은 AU도 `FAS`로 갱신 후보.

**검출 알고리즘**:
1. AN(`rob_d2_analysis`)에 `FAS`/`Full Analysis Set`/`full analysis set` (정확 매치, 대소문자 무시) 포함된 행을 모두 수집.
2. AU 현재 값이 `ITT`인 행만 후보 (이미 `FAS`인 행은 제외).
3. 부정 키워드 윈도 검사 (15자) — AN에 `not specified`/`명시 없음` 등이 FAS 매치 직후에 오면 매치 무효.

**8단계 채팅 표시 항목**: study_id, 행번호, AN 원문, 현재 AU(`ITT`), 제안값(`FAS`)

**작업자 처리**: 원문 method에서 FAS 정의가 실제로 적용되었는지 확인 후 AU 수정.

### C. KM vs WM 단독비교 행 제거 후보

v2.6.1에서 `KM_alone_vs_WM`은 본 메타분석에서 제외된다. 기존 추출 행을 검출.

**검출 알고리즘**:
1. 기본정보 Z열(`comparison_type`)이 `KM_alone_vs_WM`인 행을 모두 수집.
2. 각 행의 study_id에 대해 같은 study_id의 아웃컴 시트 행 수도 함께 카운트.
3. 3arm 연구 여부(같은 study_id에 다른 비교쌍 행이 있는지)도 표시.

**8단계 채팅 표시 항목**: study_id, 행번호, comparison_type, exclude 상태, 같은 study_id의 다른 비교쌍 유무

**작업자 처리**:
- 단순 2arm KM vs WM 연구 → `exclude=Y`로 마킹 + `exclude_reason`에 `"v2.6.1: KM vs WM 단독비교 제외"`
- 3arm 중 KM·WM·KM+WM이면 `KM vs WM` 행만 삭제, `WM vs KM+WM` 행은 보존
- 본 스킬은 행을 자동으로 삭제하지 않는다 (실수 시 복구 어려움).

### D. S열 코드 5 케이스 세분화 재검토

v2.6에서 코드 5는 "HR≥110 명시 시만"으로 좁혀졌고, v2.6.1에서 케이스 세분화로 더 정교해졌다:
- 인정: (RVR 명시 ∧ HR≥110) ∨ (RVR 미언급 ∧ HR≥110 명시)
- 불인정: RVR 단어만 있고 HR 110 미달/미언급 → T열에 원문 표현만

**검출 알고리즘**:
1. 기본정보 S열(`af_type_other`)이 `5`인 행을 모두 수집.
2. 각 행에 대해 T(`af_type_text`)/Methods 텍스트에서 다음을 검사:
   - RVR 단어 유무 (`RVR`, `rapid ventricular response`, `快速心室率`)
   - HR 임계값 매치 (`≥\s*110`, `>\s*=?\s*110`, `110\s*次/分`, `110\s*次/min`, `110\s*回/min`, `110\s*bpm`)
3. RVR은 있는데 HR 110 매치 없으면 ⚠️ "코드 5 불가 후보" 표시.
4. (v2.6에서 이미 처리된 경우라도 v2.6.1 케이스 세분화로 재검토)

**8단계 채팅 표시 항목**: study_id, 행번호, S열 값, T열 원문, RVR 매치/HR 매치 여부

**작업자 처리**: 원문 대조 후 S열 비우기 + T열에 원문 표현 보강.

### E. T열 `af_type_text` 누락 후보

v2.6.1 §1.T-01에서 T열은 자유 텍스트로 원문 AF 표현을 보존한다. T열이 공란인데 원문에 표현이 있을 가능성을 검출.

**검출 알고리즘**:
1. T열(`af_type_text`)이 공란/`NR`/`-`인 행을 수집.
2. (참고용) 같은 row의 R열(`af_type_code`)이나 S열에 코드가 있다면 원문에 표현이 있을 가능성 높음.

**8단계 채팅 표시 항목**: study_id, 행번호, R/S 코드, T 현재 값

**작업자 처리**: 원문에서 paroxysmal/persistent/permanent/RVR/non-valvular AF 등 표현 확인 후 T열 보강.

### F. 미네소타 QOL 변환 후보 — `MLHFQ` → `QOL(Minnesota cited)` 가능성

표준 MLHFQ로 추출된 행 중 **점수 방향이 표준 MLHFQ와 모순되는 케이스만** 검출 (사용자 합의: v2.6.1).

> **범위 제한**: 본 검출 룰은 **`outcome_std`가 `MLHFQ` 또는 `Minnesota Living with Heart Failure Questionnaire`인 행에만** 적용. 다른 지표(LVEF, BNP, TER, 6MWT 등)의 점수 방향 모순은 추출 단계 6B ⑤ 불확실 항목 + 작업자 검수 책임이며, 본 알고리즘 대상이 아님.

**전제: 표준 MLHFQ 점수 방향**
- 척도 범위: 0~105
- **점수가 낮을수록 환자 삶의 질 양호** (= 치료 효과 좋음)
- 따라서 효과적인 치료군은 baseline 대비 점수가 **감소**하거나, 대조군보다 **낮은 점수**를 보여야 함.

**검출 알고리즘**:

1. 아웃컴 시트에서 C열(`outcome_std`) = `MLHFQ` 또는 `Minnesota Living with Heart Failure Questionnaire`인 행을 수집.
2. 각 행에 대해 다음 자동 판별 룰을 순차 적용:

   **룰 R1 — 원시데이터 방향 검사** (H~M·S열 모두 채워진 경우):
   - 변수: `E_val1`(치료군 평균), `C_val1`(대조군 평균), `direction`(S열)
   - 모순 조건 (둘 중 하나):
     - `direction == "치료군 유리"` AND `E_val1 > C_val1` (치료군 점수가 더 높음 = 표준 방향과 어긋남)
     - `direction == "대조군 유리"` AND `E_val1 < C_val1` (대조군 점수가 더 높음 = 표준 방향과 어긋남)
   - 모순 시 ⚠️ 후보로 마킹.

   **룰 R2 — 효과크기 부호 검사** (N~Q열 채워진 경우, 원시 없거나 둘 다 검사):
   - 변수: `effect_type`(N열), `effect_value`(O열), `direction`(S열)
   - MD/SMD 가정 (E - C 형태):
     - `direction == "치료군 유리"` AND `effect_value > 0` (E가 C보다 높음 = 모순)
     - `direction == "대조군 유리"` AND `effect_value < 0` (E가 C보다 낮음 = 모순)
   - 95% CI 부호도 함께 점검 (P·Q열): CI가 0을 포함하지 않는데 부호 방향이 direction과 어긋나면 ⚠️.
   - 모순 시 ⚠️ 후보로 마킹.

   **룰 R3 — 변화량(Change) 행 별도 검사** (`time_point`이 baseline이 아닌 Change/After 행):
   - 표준 MLHFQ는 치료 후 점수 **감소**가 정상. `E_val1 > 0`(치료군 점수 증가)이면서 `direction == "치료군 유리"`면 모순. (단, 행이 절대값이 아닌 변화량으로 보고된 경우만 적용 — `outcome_original` 또는 `notes`에 "변화량" 명시 확인)

3. 룰 R1·R2·R3 중 **하나라도** 모순 매치된 행은 §F 후보로 출력.

**오탐 차단**:
- `direction`이 "차이없음" / 공란 / NaN인 행은 모순 판정 안 함 (skip).
- `E_val1` 또는 `C_val1`이 NaN/공란인 행은 R1 skip (R2만 시도).
- 모든 변수가 NaN이면 후보에서 제외 (자동 판별 불가).

**8단계 채팅 표시 항목**: study_id, 행번호, outcome_std, time_point, E_val1/C_val1, direction, effect_value, 매치된 룰(R1/R2/R3), 모순 사유(예: "direction=치료군 유리 but E_val1(67.2) > C_val1(54.8)")

**작업자 처리**: 원문 대조 후 척도가 표준 MLHFQ가 맞으면 그대로 둠 (예: 작업자가 추출 단계에서 direction 오기재한 경우 → direction만 정정), 인용 형태/방향 모순이 척도 차이에서 기인하면 `outcome_std`를 `QOL(Minnesota cited)`로 변경 + notes에 척도 의심 사유 보강.

> **참고**: 과거 회의(2026-05-19)에서 No.619·635가 알려진 사례로 거론됐으나, 본 알고리즘에서는 study_id 우선순위를 별도로 두지 않는다. 두 논문 모두 점수 방향 모순으로 자동 검출되어 후보에 포함된다.

### G. 3arm 비교쌍 점검

기본정보에 같은 study_id가 2회 이상 등장하는 케이스를 표시 (3arm·복수 비교쌍 가능성).

**검출 알고리즘**: study_id 카운트, 2회 이상이면 후보.

**8단계 채팅 표시 항목**: study_id, 행번호 목록, 각 행의 comparison_type

**작업자 처리**: v2.6.1 §1.0-01 룰에 따라 비교쌍 선택 — KM·WM·KM+WM 3군 구성이면 `WM vs KM+WM`만 유지하고 `KM vs WM`은 제거.

---

## 멱등성 보장

- 자동 처리는 모두 정확 매치 + 이미 적용된 상태 감지로 두 번 실행 시 변경 0건.
- `spec_version` 갱신은 현재 값이 이미 `2.6.1`이면 skip.
- 사람 확인 항목은 8단계 인터랙티브에서 처치 후 검출 알고리즘에서 자연히 제외 (예: KM_alone_vs_WM이 `exclude=Y`로 마킹되면 §C 후보에서 빠짐, ITT→FAS로 갱신되면 §B 후보에서 빠짐) → 두 번째 실행 시 멱등.
- 작업자가 8단계에서 "건너뛰기"한 항목은 다음 세션 재실행 시 다시 채팅에 등장 — `.md` 리포트의 미처리 목록과 일치해 추적 가능.

## v2.6과의 관계

- v2.5.x 마스터는 v2.6 명세를 먼저 적용 후 v2.6.1 명세를 순차 적용.
- v2.6 마스터는 v2.6.1 명세만 적용.
- v2.6.1 마스터는 변경 0건으로 종료.
